# TaelgarVerse Radical Improvement Ideas

Prepared: 2026-05-29

This is a corrected brainstorm for larger reading-experience and navigation ideas. It replaces the earlier version's bad assumptions: do not invent a new metadata schema, do not require YAML authoring for generated content, and do not add large amounts of frontmatter when the vault already has mature metadata and Markdown-generation conventions.

## Ground Rules

- Respect the Taelgar vault metadata system as the source of truth. Relevant references live in `_MoC/`, especially `Metadata Specification.md`, `Display Control.md`, `Note Categorization.md`, and the related data-quality notes and `.base` files.
- Do not introduce a competing entity schema. Use existing concepts such as `tags`, `typeOf`, `whereabouts`, `affiliations`, `campaignInfo`, `knownTo`, `partOf`, `displayDefaults`, `name`, `aliases`, session metadata, and the existing display-token system.
- Do not make rich generated content depend on hand-editing YAML. Obsidian editing should stay Markdown-first.
- Keep frontmatter additions minimal. If a page needs generation control, prefer existing fields such as `sessionKey`, `session-template`, `template`, `hide`, `excludePublish`, and existing session fields before adding anything new.
- Prefer generated Markdown slots over embedded HTML or new frontmatter blobs.
- Any generated source fragments should remain excluded from publication unless intentionally surfaced.

## Existing Pattern To Build On

The `Into the Chasm` generated session-note flow is the useful model:

- A published session note contains normal, editable Markdown plus modest existing frontmatter.
- The note identifies a generated session key and template.
- Excluded `_generated/session-notes/...` fragments contain Markdown slots marked with HTML comments.
- A source template uses `{slot.name}` placeholders to compose the final Markdown.
- Generated fragments can contain rich Markdown, wiki links, callouts, Dataview-style inline fields, and display-token placeholders without requiring the user to edit YAML.

Representative slot pattern:

```md
<!-- SLOT: narrative.short -->
Short Markdown summary text.
<!-- /SLOT -->

<!-- SLOT: narrative.intermediate -->
Medium Markdown summary text.
<!-- /SLOT -->

<!-- SLOT: narrative.long -->
Long Markdown summary text.
<!-- /SLOT -->
```

That pattern should be the default design constraint for richer session pages and generated context.

## Zoomable Session Notes

The target is closer to the attached `lost-in-the-feywild-002-summary.html` example:

- A page-wide zoom control with levels such as `Short`, `Intermediate`, `Long`, and `Transcript`.
- A compact beat/scene navigation list.
- Each beat has the same stable heading and id across all zoom levels.
- The selected zoom level changes the amount of text shown for every beat.
- Transcript view is another zoom layer, not a separate wall of text disconnected from the summary.
- The reader can start with one sentence per beat and deepen progressively.

Conceptually, the generated page has this shape:

```md
# Session Title

Intro/one-paragraph session summary.

<zoom controls generated by the website layer>

<beat nav generated from beat slots>

## B01 Through the Root

::: zoom short
One-sentence beat summary.
:::

::: zoom intermediate
One-paragraph beat summary.
:::

::: zoom long
Full beat summary.
:::

::: zoom transcript
Transcript lines for this beat.
:::
```

The authoring source should not literally require hand-written HTML. Better options:

- Generate those blocks from Markdown slots.
- Add a small exporter transform that recognizes zoomable session blocks and emits the HTML/CSS/JS structure.
- Keep the source readable in Obsidian even if the interactive website behavior is richer.

Possible slot model:

```md
<!-- SLOT: beats.nav -->
- B01: Through the Root
- B02: Bodies on the Road
<!-- /SLOT -->

<!-- SLOT: beat.B01.title -->
Through the Root
<!-- /SLOT -->

<!-- SLOT: beat.B01.short -->
The party loses contact with Holda and crawls through the root into the Twilight Kingdom.
<!-- /SLOT -->

<!-- SLOT: beat.B01.intermediate -->
Still crawling through the narrow root tunnel, the party tries to reach Holda by message...
<!-- /SLOT -->

<!-- SLOT: beat.B01.long -->
The night resumes in the root tunnel beneath Dandelion House...
<!-- /SLOT -->

<!-- SLOT: beat.B01.transcript -->
- **DM:** ...
- **Ayveen:** ...
<!-- /SLOT -->
```

Questions for the real design:

- Should the website transform consume individual `beat.*` slots directly, or should the session-note generator produce a single assembled `zoomable.narrative` slot?
- Should transcript lines be rendered as Markdown lists, HTML generated by the exporter, or a custom fenced/code block transformed by the website layer?
- Should the selected zoom level persist per site, per campaign, or per page?
- Should search index only the long summaries, or also transcript text when transcript is public?

Good first prototype:

- Use one generated session from `Lost in the Feywild` or `Into the Chasm`.
- Generate beat slots for short/intermediate/long/transcript.
- Add one source template variant for zoomable sessions.
- Add only the minimum website JS/CSS needed to reproduce the attached example's behavior inside MkDocs Material.
- Do not change historical sessions in bulk until the prototype is good.

## Pull Quotes

Pull quotes are useful, but they should be another generated Markdown slot surface, not a new metadata schema.

Possible slot pattern:

```md
<!-- SLOT: beat.B04.quotes -->
> "A short memorable line."
> - Speaker, B04
<!-- /SLOT -->
```

Display options:

- Quote block inside the relevant beat.
- Optional "Notable lines" section near the top of the session.
- Campaign landing-page excerpt from recent sessions.

Guardrails:

- Keep quote selection explicit or generated into reviewable Markdown.
- Do not publish long transcript excerpts casually.
- If audio clips are later attached, align them by generated Markdown text or source transcript ids rather than hand-authored frontmatter.

## Multimedia

Richer media should use normal Markdown/source slots first:

```md
<!-- SLOT: session.media -->
![[dandelion-house-map.webp]]

![[lord-endless-ending-banner.webp]]

![[audio/endless-ending-clock-strikes-zero.mp3]]
<!-- /SLOT -->
```

Possible website improvements:

- Banner treatment for a session or campaign page when a normal Markdown image is present in the right slot.
- Gallery layout for a generated `session.media` slot.
- Audio controls for clipped session recordings, using the existing audio embed work rather than inventing a new format.

Constraints:

- Asset paths and embeds should continue to flow through the exporter asset index.
- Large media must respect the existing asset-reporting and optimization pipeline.
- Media generation should not require new YAML blocks.

## Metadata-Driven Context

The earlier brainstorm was wrong to propose a new entity schema. The right direction is to build generated context on the existing vault metadata and display managers.

Existing metadata already covers:

- Note identity and categorization: `tags`, `typeOf`, `typeOfAlias`, `species`, `ancestry`, `name`, `aliases`.
- Time: `born`, `created`, `died`, `destroyed`, `DR`, `DR_end`, `realWorldDate`.
- Relationships: `whereabouts`, `affiliations`, `campaignInfo`, `knownTo`, `partOf`.
- Display behavior: `displayDefaults` and the token format system.
- Session notes: `campaign`, `sessionNumber`, `realWorldDate`, `DR`, `DR_end`, `players`, `companions`, `descTitle`, `tagline`.
- Publication/display controls: `excludePublish`, `hide`, `template`, `activeYear`.

Future generated context should use this data rather than duplicating it.

Examples:

- Person pages can use existing `knownTo`, `campaignInfo`, backlinks from session notes, `whereabouts`, and `affiliations` to generate campaign context.
- Place pages can use existing `whereabouts` chains and type categorization to generate region breadcrumbs.
- Session pages can use existing session metadata and generated cast/location slots to orient readers.
- Index pages can use existing tags/type fields and display-token output for summaries.

## Generated "Appears In" Sections

This remains a good idea, but it should be framed as a derived output from existing metadata and session-note links.

Possible generated Markdown slot:

```md
<!-- SLOT: generated.appears_in -->
### Appears In

- [[Into the Chasm - Episode 02]]: questioned by the party in Zeyfa's Labyrinth.
- [[Into the Chasm - Episode 03]]: remembered in discussion of Melusa.
<!-- /SLOT -->
```

Or more likely this could be a dataview or dataviewJS query? 

Potential sources:

- Session-note backlinks.
- Existing `campaignInfo` and `knownTo`.
- Generated session cast/location/item slots.
- Beat-facts/session pipeline output, if it is already generating reviewable Markdown.

Important distinction:

- Backlinks answer "what links here?"
- `Appears In` should answer "where did this entity matter in play?"

This should probably start as generated Markdown appended or slotted into selected pages, not a live client-side graph.

## Cards and Index Pages

Cards and index pages can be useful when they are generated from existing metadata, but should not become decorative page chrome.

Good candidates:

- Campaign cards: current/archive status, party page, latest session, session count.
- Person index cards: display name, type/species/ancestry, status if already represented, known-to campaign, current whereabouts.
- Place index cards: type, parent location/region from whereabouts, short description if present.
- Item/treasure cards: image, current owner/location, rarity or mechanics link when already present.
- Session cards: campaign, session number, date, tagline, short summary.

Implementation direction:

- Generate static Markdown card grids first.
- Use existing display-token logic for human-readable labels.
- Add client-side filtering later only if static indexes are not enough.
- Exclude duplicate generated index pages from search if they hurt result quality.

## Breadcrumb and Context Text

Generated breadcrumbs/context lines should reuse display data rather than new fields.

Examples:

- `Person known to the Dunmar Fellowship; currently in Roscombe, Addermarch.`
- `Settlement in Roscombe, Addermarch.`
- `Into the Chasm, Episode 02; played December 3, 2025; set on DR 1730-01-25.`

Implementation direction:

- Use NameManager/display-token output where possible.
- Generate only when data is present and trustworthy.
- Keep missing-data behavior silent; do not show empty labels.
- Add warnings for unresolved metadata references rather than inventing fallback schema.

## Prototype Plan

### Prototype 1: Zoomable Session Page

- Pick one `Lost in the Feywild` session because the attached example already shows the desired shape.
- Produce generated Markdown beat slots for short/intermediate/long/transcript.
- Create a template variant that inserts a single zoomable narrative block.
- Add website-layer CSS/JS matching the example's behavior inside MkDocs Material.
- Keep the source session note Markdown-readable.

### Prototype 2: Generated Appears-In Slot

- Pick a small set of NPCs from one campaign.
- Generate an excluded Markdown fragment per page or per campaign with `generated.appears_in` slots.
- Source appearances from session links plus existing campaign metadata.
- Insert the slot into a page template or append it during export.
- Review for false positives before scaling.

### Prototype 3: Static Rich Index

- Pick one domain, probably People or Sessions.
- Generate a static Markdown index using existing metadata and display output.
- Use compact cards only where they improve scanning.
- Do not add filters until the static page proves useful.

## Non-Goals

- Do not invent a new universal metadata schema.
- Do not require manual YAML editing for rich session content.
- Do not bulk-add frontmatter to thousands of notes.
- Do not make the exporter infer too much from brittle paths when tags/metadata/slots can carry the intent.
- Do not build client-side graph/search/filter machinery before the generated Markdown version is useful.

## Near-Term Recommendation

Replace the vague "radical improvement" idea with a concrete zoomable session-note prototype based on the attached HTML and the existing generated-slot workflow. The first implementation should prove the rendering model, authoring model, and search behavior on one session. After that, generated `Appears In` sections and richer indexes can build from the same existing metadata and session-slot data.
